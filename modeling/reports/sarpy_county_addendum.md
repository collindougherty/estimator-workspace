## Comparison with Douglas County source
Douglas source (`omaha_properties_raw_master.csv`) is materially richer for the current pipeline:
- **216,640 rows / 64 columns** vs. Sarpy **76,882 / 42**
- Douglas already has canonical split address fields (`property_a`, `house`, `street_dir`, `street_nam`, `street_typ`)
- Sarpy requires parsing `siteaddress` into canonical address components
- Douglas has assessor `quality`, `condition`, `numbldgs`, and reliable `bldg_story`
- Sarpy does **not** provide `quality`, `condition`, or `numbldgs`; `bldg_story` is mostly unavailable because `resstrtyp` is usually a structure type code rather than a numeric story count
- Sarpy also lacks a direct lot square-foot field, so staged `sq_feet` is derived as `acreage * 43,560`

Selected canonical completeness comparison (`sarpy_vs_douglas_canonical_comparison.csv`):
- `quality`: Douglas **88.7%**, Sarpy **0%**
- `condition`: Douglas **88.7%**, Sarpy **0%**
- `numbldgs`: Douglas ~**100%**, Sarpy **0%**
- `bldg_story`: Douglas ~**100%**, Sarpy **3.6%**
- `bldg_yrblt`: Douglas ~**100%**, Sarpy **80.8%**
- `street_typ`: Douglas **95.1%**, Sarpy **64.8%**

## How Sarpy was normalized for the existing modeling schema
New script: `modeling/scripts/stage_sarpy_county.py`

Canonical mapping used in the staged copy:
- `parcelid -> pin`
- parsed `siteaddress -> property_a/house/street_dir/street_nam/street_typ`
- `pstlcity` + `siteaddress` suffix parsing -> `prop_city`
- `pstlzip5` + `siteaddress` suffix parsing -> `prop_zip`
- `lndvalue -> land_value`
- `impvalue -> improvemen`
- `cntassdval -> total_valu`
- `schldscrp -> school_dis`
- `resflrarea -> bldg_sf`
- `resyrblt -> bldg_yrblt`
- `acreage -> acres`
- derived `sq_feet = acreage * 43,560`
- derived `class = 'R'` when Sarpy description contains `RESIDENTIAL`, else `NONR`

Extra Sarpy provenance columns were preserved in the staged CSV (`source_county`, `sarpy_classcd`, `sarpy_classdscrp`, `sarpy_usedscrp`, `sarpy_raw_siteaddress`, etc.) so later debugging does not need to revisit the external raw file.

## How it should be joined / used in the modeling pipeline
New script: `modeling/scripts/prepare_multicounty_dataset.py`

Current recommended baseline flow:
1. Stage Sarpy once into canonical form:
   ```bash
   python3 modeling/scripts/stage_sarpy_county.py
   ```
2. Build a permit/property join across both counties:
   ```bash
   python3 modeling/scripts/prepare_multicounty_dataset.py \
     --permit-csv modeling/data/raw/roofing/residential_reroof_2025-11-01_2026-04-14.csv \
     --property-source douglas=/Users/collindougherty/MyCloud/properties/omaha_properties_raw_master.csv \
     --property-source sarpy=modeling/data/raw/properties/sarpy_county/sarpy_properties_model_ready.csv
   ```
3. Use the multicounty outputs under `modeling/data/processed/multicounty/` for downstream model training once permit windows actually include Sarpy parcels.

The join remains **parcel-number first** (`permit parcel_number -> canonical pin`). That is the safest low-compute default.

## Data-quality caveats
- `siteaddress` is only populated for **92.5%** of Sarpy rows.
- Mailing city can differ from site city, so property location should prefer parsed `siteaddress` over `pstlcity` when available.
- `resstrtyp` is not a reliable analog to Douglas `bldg_story`; only numeric values were retained.
- `quality`, `condition`, and `numbldgs` are absent in Sarpy and should either be left missing with indicators or replaced by county-aware imputations.
- Derived `class` in Sarpy is intentionally coarse (`R` vs `NONR`); it is suitable for the current residential cohort filter, but not a perfect county-to-county tax-class mapping.
- Derived `sq_feet` reflects lot size from acreage, not an assessor-provided lot-square-foot field.

## Recommended next steps for a combined-county baseline
1. Keep the current Douglas baseline unchanged; use the multicounty prep script only for additive experiments.
2. When a permit window includes Sarpy parcels, train from `modeling/data/processed/multicounty/*_cohort.csv` rather than the Douglas-only cohort.
3. Add explicit county/source features (`source_county`, city, ZIP) so the model can learn geographic price shifts.
4. Add missingness indicators for Sarpy-only gaps (`quality`, `condition`, `numbldgs`, `bldg_story`).
5. If Sarpy permits arrive without parcel IDs, add a separate address-crosswalk/fuzzy-address step rather than weakening the default parcel join.
6. Consider staging a full Douglas canonical copy later if a single canonical property master is preferred for repeated multicounty experiments.
